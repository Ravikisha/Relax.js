\chapter{Attributes, properties, classes, and styles (\texttt{src/attributes.ts})}
\label{ch:attributes-props}

In Chapter~\ref{ch:mounting} we learned how to mint DOM nodes. This chapter is about the part that looks boring in demos but dominates correctness in real apps:
\emph{wiring props into the DOM in a way that matches how the browser actually works}.

I call this the \textbf{DOM edge}. It's where frameworks either:

\begin{itemize}
  \item accumulate a huge and fragile compatibility layer, or
  \item pick a small set of rules and defend them with tests.
\end{itemize}

Relax chooses the second path. The module \texttt{src/attributes.ts} is intentionally tiny, and the ``spec'' is mostly the test matrix in \texttt{src/\_\_tests\_\_/attributes.test.ts}.

\section{The mental model: the browser has \emph{two} prop channels}

When people say ``set an attribute'', they often mean three different things:

\begin{enumerate}
  \item \textbf{Attributes}: string key/value pairs in the element's attribute map (example: \texttt{el.getAttribute('id')}).
  \item \textbf{Properties}: live fields on the element object (example: \texttt{input.value}).
  \item \textbf{Special APIs}: \texttt{className}/\texttt{classList} and \texttt{style} are their own worlds.
\end{enumerate}

The key renderer lesson is that \textbf{attributes and properties are not interchangeable}. The classic foot-gun is forms:
	\texttt{<input value="x">} and \texttt{input.value = "x"} look similar, but behave differently across updates.

Relax's rule of thumb is extremely pragmatic:

\begin{quote}
	\textbf{Relax rule:} only \texttt{data-*} is forced through the attribute channel. Everything else is treated as a property.
\end{quote}

This rule is small enough to memorize, and it keeps the ``boring'' cases correct by default.

\section{Contract: \texttt{setAttributes(el, attrs)}}

In Relax, element props are written by one function:

\begin{quote}
	\texttt{setAttributes(el: HTMLElement, attrs: ElementVNodeProps)}
\end{quote}

You saw calls to this during element mounting in Chapter~\ref{ch:mounting}. You'll see it again during patching in Chapter~\ref{ch:patching-diffing}.

\subsection*{Inputs}
\begin{itemize}
  \item \texttt{el}: a real DOM element you already created.
  \item \texttt{attrs}: the element props object from the VNode (which may contain \texttt{class}, \texttt{style}, \texttt{key}, and many DOM-ish names).
\end{itemize}

\subsection*{Outputs}
\begin{itemize}
  \item The DOM element is mutated in-place.
  \item No return value; this is a pure ``effect'' boundary.
\end{itemize}

\subsection*{Non-obvious invariants}
\begin{itemize}
  \item \textbf{\texttt{key} never reaches the DOM.} It's identity metadata only.
  \item \textbf{\texttt{class} and \texttt{style} are special-cased.} They do not go through the generic channel.
  \item \textbf{Most props become properties, not attributes.} This is a correctness-first bias.
\end{itemize}

\subsection*{The implementation (real code)}
This is \texttt{setAttributes()} from \texttt{src/attributes.ts}. Notice the three-stage pipeline: split special fields, delete \texttt{key}, then apply the rest.

\begin{lstlisting}[language=JavaScript]
export function setAttributes(el: HTMLElement, attrs: ElementVNodeProps) {
  const { class: className, style, ...otherAttrs } = attrs

  // Delete the "key" property if it exists
  delete (otherAttrs as any).key

  if (className) {
    setClass(el, className)
  }

  if (style) {
    Object.entries(style).forEach(([prop, value]) => {
      setStyle(el, prop, value)
    })
  }

  for (const [name, value] of Object.entries(otherAttrs)) {
    setAttribute(el, name, value as any)
  }
}
\end{lstlisting}

That deletion might look redundant, because other modules also delete \texttt{key}. This redundancy is a feature.
I want every layer that can touch ``props'' to be safe even if you use it in isolation.

\section{The core decision: \texttt{setAttribute(el, name, value)}}

The entire attribute/property policy collapses into one small function:

\begin{lstlisting}[language=JavaScript]
export function setAttribute(el: Element, name: string, value: string | number | null) {
  if (value == null) {
    removeAttribute(el, name)
  } else if (name.startsWith('data-')) {
    el.setAttribute(name, String(value))
  } else {
    ;(el as any)[name] = value
  }
}
\end{lstlisting}

There's an important design choice hiding here:

\begin{itemize}
  \item Relax does \emph{not} try to maintain a massive allowlist of ``attribute names''.
  \item Instead, it assumes \textbf{``property first''} and relies on browser behavior.
\end{itemize}

This is why Relax feels predictable: most names you type behave like real DOM fields.

\section{Contract: what does ``property first'' buy us?}

If you want to understand this module quickly, don't start by reading framework wars on the internet.
Start by reading the invariants encoded in \texttt{src/\_\_tests\_\_/attributes.test.ts}.

Here are a few highlights that act like a minimal spec:

\begin{itemize}
  \item \textbf{Class replacement semantics:} setting \texttt{class} again replaces the old value.
  \item \textbf{Style is assignment-based:} setting \texttt{style} twice accumulates fields unless you explicitly remove them.
  \item \textbf{Form controls behave like fields:} \texttt{input.value}, \texttt{input.checked}, \texttt{select.value} follow DOM property rules.
  \item \textbf{Null means remove:} writing \texttt{null} maps to removal behavior (with element-specific defaults).
\end{itemize}

For example, the test suite proves that \texttt{tabIndex} removal yields \texttt{-1} (DOM default), and that a \texttt{<select>} value that doesn't match an option becomes \texttt{''}.

\section{Removal is not symmetric: \texttt{removeAttribute(el, name)}}

Removing looks trivial until you meet element-specific behavior and host quirks.
Relax uses a two-level cleanup:

\begin{lstlisting}[language=JavaScript]
export function removeAttribute(el: Element, name: string) {
  try {
    ;(el as any)[name] = null
  } catch {
    if (typeof console?.warn === 'function') {
      console.warn(`Failed to set "${name}" to null on ${(el as any).tagName}`)
    }
  }

  el.removeAttribute(name)
}
\end{lstlisting}

This is one of those places where being ``correct'' means respecting the platform.
Some DOM properties throw when assigned \texttt{null} (or behave differently between real browsers and jsdom), so I prefer a best-effort property clear plus a guaranteed attribute removal.

\section{Class and style are special-cased (on purpose)}

There are pragmatic reasons to split out \texttt{class} and \texttt{style}:

\begin{itemize}
  \item \textbf{\texttt{class}:} I want a stable, replacement-based behavior and I want arrays to work.
  \item \textbf{\texttt{style}:} I want to assign fields directly (it matches CSSOM), and I don't want to build a diff engine yet.
\end{itemize}

The tests at the top of \texttt{src/\_\_tests\_\_/attributes.test.ts} act as the contract for both.

\subsection*{A subtle style invariant}
	\texttt{setAttributes()} doesn't know the previous style object.
So a later call like \texttt{\{ style: \{ backgroundColor: 'blue' \} \}} will \emph{not} remove an earlier \texttt{color} field.
That's not a bug; that's a design constraint.
In Chapter~\ref{ch:patching-diffing}, you'll see how patching handles this by diffing old vs new props.

\section{Props vs events: keeping the shape sane}\label{sec:props-vs-events}

Relax stores event handlers under \texttt{props.on}.
Before mounting an element or instantiating a component, we split the object into two:

\begin{quote}
	\texttt{extractPropsAndEvents(vdom) -> \{ props, events \}}
\end{quote}

This is the real implementation from \texttt{src/utils/props.ts}:

\begin{lstlisting}[language=JavaScript]
export function extractPropsAndEvents(vdom: { props: Record<string, unknown> }) {
  const { on: events = {}, ...props } = vdom.props as any
  delete (props as any).key
  return { props, events }
}
\end{lstlisting}

Again, note the invariant: \texttt{key} is consumed by the renderer and must never leak into the DOM or component props.

\subsection*{Invariant checklist (props/events split)}
The tests in \texttt{src/\_\_tests\_\_/props.test.ts} prove five simple facts:

\begin{itemize}
  \item empty props \textrightarrow{} empty \texttt{props}
  \item missing \texttt{on} \textrightarrow{} empty \texttt{events}
  \item \texttt{key} is ignored
  \item non-event fields stay in \texttt{props}
  \item \texttt{on: \{...\}} becomes \texttt{events}
\end{itemize}

When you're building systems like this, these are the invariants you defend.
Everything else is just API surface.

\section{Design notes: why Relax stays small here}

The attribute layer is a great place to accidentally create a maintenance nightmare.
Relax deliberately keeps the rule set tiny:

\begin{itemize}
  \item property-first by default
  \item explicit \texttt{data-*} rule
  \item explicit \texttt{class}/\texttt{style} behavior
  \item tests as a compatibility net
\end{itemize}

If you were extending Relax, the first realistic feature requests would be:

\begin{itemize}
  \item \textbf{SVG support}: namespaces and attribute/property differences.
  \item \textbf{Style removal semantics}: tracking old style keys (or letting patching own it entirely).
  \item \textbf{Boolean attribute reflection}: some teams prefer attribute-presence semantics.
\end{itemize}
