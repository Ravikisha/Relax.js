\chapter{HRBR Blocks: Template + Slots (the Compiled Fast Path)}
\label{ch:hrbr-blocks-template-slots}

Chapters 12 and 13 gave us two halves of a reactive UI runtime:

\begin{itemize}
  \item \textbf{Signals} decide \emph{what} values changed.
  \item \textbf{Scheduler lanes} decide \emph{when} to apply work.
\end{itemize}

Blocks are the third half: they decide \emph{how} to touch the DOM.

A \textbf{block} is the "fast DOM" path: it describes a stable DOM template and a set of precise write locations (slots). Updates become \emph{O(k)} slot writes instead of building a VDOM tree and diffing children.

\begin{quote}
	extbf{Blocks replace diffing with addressing.} The template fixes structure. Slots give exact addresses. Updates become a handful of targeted DOM writes.
\end{quote}

\section{Why blocks exist (what VDOM cannot do efficiently)}

VDOM diffing is a \emph{general-purpose} way to update UI.
That generality costs work:

\begin{itemize}
  \item allocate a new tree of VNodes
  \item compare old/new trees
  \item figure out where the DOM should change
\end{itemize}

Compiled UI wants the opposite trade-off: specialize up-front so runtime work is minimal.
Blocks do that by encoding two things that VDOM typically learns at runtime:

\begin{itemize}
  \item \textbf{Structure is stable}: the template defines the DOM shape.
  \item \textbf{Dynamic writes are explicit}: slots pre-compute where updates go.
\end{itemize}

The result is a small set of direct DOM operations per update.
This is the same general idea behind other compiler-first frameworks.
Svelte has a well-known write-up for the intuition (even if you disagree with the framing, the mental model is useful):
\href{https://svelte.dev/blog/virtual-dom-is-pure-overhead}{https://svelte.dev/blog/virtual-dom-is-pure-overhead}.

\section{Minimal block (end-to-end)}

Let's build the smallest meaningful block: a greeting with a dynamic name and a click handler.
This is not pseudo-code; it matches the contracts in \texttt{runtime/block.ts}.

\begin{lstlisting}[language=JavaScript]
import { defineBlock, mountBlock } from './runtime/block'

const HelloBlock = defineBlock({
  templateHTML: `<div>Hello <span>world</span> <button>ok</button></div>`,
  slots: {
    // root.childNodes = [Text('Hello '), span, Text(' '), button]
    // span.childNodes = [Text('world')]
    name: { kind: 'text', path: [1, 0] },

    // button is childNodes[3]
    onClick: { kind: 'event', path: [3], name: 'click' },
  },
})

const host = document.getElementById('app')
const mounted = mountBlock(HelloBlock, host, {
  name: 'Ravi',
  onClick: () => console.log('clicked'),
})

// later: this is O(1) targeted DOM writes
mounted.update({ name: 'Relax' })

// cleanup
mounted.destroy()
\end{lstlisting}

Two details matter more than they look:

\begin{itemize}
  \item The \texttt{text} slot must point to an \emph{existing} Text node in the template. That's why the template uses \texttt{<span>world</span>} instead of \texttt{<span><!--placeholder--></span>}.
  \item The \texttt{path} walks \texttt{childNodes}, not \texttt{children}. Text nodes (including whitespace) count.
\end{itemize}

\section{A tiny contract (what a block guarantees)}

Treat this as the “promise” the runtime makes to compiled code:

\begin{enumerate}
  \item \textbf{Single stable root:} \texttt{templateHTML} must produce exactly one root element, which is cloned and appended to the host.
  \item \textbf{Exact addressing:} each slot points to a node using a \texttt{path: number[]} walk over \texttt{childNodes} indices.
  \item \textbf{Specialized patching:} slot kinds are patched by small, explicit routines (text/attr/prop/class/style/event).
  \item \textbf{No needless DOM writes:} redundant values are skipped via \texttt{Object.is}.
  \item \textbf{Safe lifecycle:} \texttt{destroy()} is idempotent, removes listeners, and after destroy further \texttt{update()} calls are ignored.
\end{enumerate}

\section{Block vocabulary}

\subsection{TemplateHTML}

A block definition includes a \texttt{templateHTML: string}.
This must produce exactly one root element.

Implementation detail: \texttt{parseTemplate(templateHTML)} uses a \texttt{<template>} element and asserts:

\begin{quote}
\texttt{templateHTML must have a single root element}
\end{quote}

The implementation is intentionally strict:

\begin{itemize}
  \item it does \texttt{templateHTML.trim()} before parsing
  \item it reads \texttt{tpl.content.firstElementChild}
  \item it asserts non-null and returns the root element
\end{itemize}

This makes “multiple root nodes” a loud error, which is what you want for compiled templates.

\subsection{Slot}

A \textbf{slot} is a metadata record describing \emph{where} and \emph{how} to write a dynamic value into a template instance.

Slots are keyed by string (slot key) and come in these kinds:

\begin{itemize}
  \item \texttt{text}: write to a Text node
  \item \texttt{attr}: set/remove attributes
  \item \texttt{prop}: set DOM properties
  \item \texttt{class}: write \texttt{class} attribute
  \item \texttt{style}: write inline styles
  \item \texttt{event}: bind events once and swap handler references
\end{itemize}

Each slot includes a \texttt{path: number[]} which is a \textbf{childNodes index walk} from the template root.

Two practical consequences:

\begin{itemize}
  \item \texttt{childNodes} counts text nodes and comments, so whitespace in templates matters.
  \item Slot paths are best generated by a compiler that controls the template serialization.
\end{itemize}

\section{Why paths (and not querySelector)?}

Paths are the “compiled fast path” address:

\begin{itemize}
  \item they’re \emph{exact} (no ambiguities)
  \item they’re cheap (depth walk, no subtree scan)
  \item they enable prefix caching across multiple slots
\end{itemize}

One subtle point the tests implicitly enforce: \texttt{childNodes} includes text nodes (including whitespace). That means a compiler must be in control of template serialization and whitespace if it wants stable paths.

\subsection{Path stability: whitespace is part of your ABI}

This is worth slowing down for, because it's the most surprising part of the block model.

	extbf{Paths are an ABI between the compiler and runtime.}
If the compiler emits \texttt{path: [3, 0]}, it is assuming a specific \texttt{childNodes} layout.

That means these two templates are \emph{not equivalent} from the runtime's perspective:

\begin{lstlisting}[language=JavaScript]
// Template A
`<div><span>hi</span><button>ok</button></div>`

// Template B (note the whitespace/newlines)
`<div>
  <span>hi</span>
  <button>ok</button>
</div>`
\end{lstlisting}

Template B contains additional text nodes for the whitespace/newlines.
So a slot path that points to the \texttt{button} in template A may point to a whitespace Text node in template B.

\paragraph{Practical rule for compilers}

Compilers should either:

\begin{itemize}
  \item serialize templates in a whitespace-stable canonical form, or
  \item generate paths from the exact serialized string they ship as \texttt{templateHTML}
\end{itemize}

Otherwise you'll get "works in dev, breaks in prod" bugs when minifiers/formatters rewrite whitespace.

\section{The public API (surface area)}

The core exports in \texttt{runtime/block.ts} are:

\begin{itemize}
  \item \texttt{defineBlock(def)}: identity helper for typing and readability
  \item \texttt{mountBlock(def, host, initialValues?, options?) -> MountedBlock}
  \item \texttt{mountCompiledBlock(def, host, compiledSlots, options?) -> MountedBlock \& \{ dispose() \}}
  \item \texttt{mountNestedBlock(parent, slotKey, def, initialValues?)}
  \item \texttt{mountNestedFallback(parent, slotKey, mount)}
  \item \texttt{resolvePath(root, path)} and \texttt{resolvePathCached(root, path, slotKey, dev?)}
\end{itemize}

Reading guide:

\begin{itemize}
  \item \texttt{mountBlock} is the non-reactive primitive (mount + \texttt{update}).
  \item \texttt{mountCompiledBlock} wires signals into \texttt{update} with effects and the HRBR scheduler.
  \item \texttt{mountNestedBlock}/\texttt{mountNestedFallback} are the structural escape hatches.
\end{itemize}

\section{defineBlock(): the type boundary}

	texttt{defineBlock(def)} doesn’t transform input. It exists so "compiled" code can be explicit and type-safe.

In \texttt{runtime/block.ts}, it is literally an identity function:

\begin{quote}
	texttt{export function defineBlock(def: BlockDef): BlockDef \{ return def \}}
\end{quote}

That sounds trivial, but it creates a clean “compiler boundary” and a stable type surface.

Tests use it in every case:

\begin{verbatim}
const block = defineBlock({ templateHTML: ..., slots: ... })
\end{verbatim}

\section{mountBlock(): instantiate template + resolve slot nodes}

\texttt{mountBlock()} does the non-reactive mount:

\begin{enumerate}
  \item parse template HTML into a root element
  \item clone it to create an instance
  \item append instance root to the host
  \item resolve each slot \texttt{path} into a concrete DOM \texttt{Node}
  \item apply \texttt{initialValues} via \texttt{block.update(initialValues)}
\end{enumerate}

The key optimization is “resolve once, write many”: paths are resolved at mount time and stored in \texttt{slotNodes}. Updates never traverse the DOM tree; they dereference and write.

\begin{itemize}
  \item \texttt{mountBlock} stops after returning \texttt{update()}.

\end{itemize}

The returned \texttt{MountedBlock} includes:

\begin{itemize}
  \item \texttt{root}: the mounted root element
  \item \texttt{slotNodes}: mapping from slot key to resolved node reference
  \item \texttt{update(values)}: apply slot writes
  \item \texttt{destroy()} / \texttt{dispose()} lifecycle
\end{itemize}

\subsection{Path resolution and caching}

\texttt{resolvePath(root, path)} is the simple implementation.
But \texttt{mountBlock} uses \texttt{resolvePathCached(instanceRoot, path, slotKey, dev)}.

The cache is a \texttt{Map<string, Node>} stored on the instance root as \texttt{\_\_hrbrPathCache}.
Prefixes are stored as dot-joined strings like \texttt{"1.0"}.

The caching algorithm is incremental: as it walks the path, it caches each prefix result (\texttt{"1"}, then \texttt{"1.0"}, etc.). Multiple slots that share a prefix benefit immediately.

The update loop iterates over provided keys only. If a key doesn’t exist in \texttt{def.slots}, it is ignored.

Two important performance rules:

\begin{itemize}
  \item The block stores \texttt{prevValues[key]} and skips redundant writes using \texttt{Object.is}.
  \item It emits a devtools event \texttt{slotWrite} for each actual write.
\end{itemize}

The use of \texttt{Object.is} is deliberate: it treats \texttt{NaN} as equal to itself and provides a cheap “same reference” check for objects (like style objects or class arrays).

It also affects correctness in subtle-but-useful ways:

\begin{itemize}
  \item \textbf{Stable function identity:} if a compiled block passes the same handler function reference repeatedly, the event slot can skip work.
  \item \textbf{Object slots:} if style/class are represented as objects/arrays, reusing the same value prevents unnecessary DOM churn.
\end{itemize}

Spec: \texttt{skips redundant writes for unchanged values (text + attr)}.

\subsection{text slots}

Text slots must resolve to a real Text node. The runtime asserts:

\begin{quote}
\texttt{slot 'x' expected a Text node}
\end{quote}

Writes use \texttt{Text.nodeValue} and normalize nullish values to the empty string.

Spec: \texttt{mounts a template and updates text slots by path}.

\paragraph{DOM ops (what actually happens)}

This is conceptually equivalent to:

\begin{lstlisting}[language=JavaScript]
textNode.nodeValue = next == null ? '' : String(next)
\end{lstlisting}

No nodes are created and no subtree scans happen.

\paragraph{Gotcha: Text slots are strict about node type}

The runtime asserts \texttt{nodeType === Node.TEXT\_NODE}.
So if a compiler points a text slot at an Element (for example, thinking in terms of \texttt{textContent}), updates will throw.
This strictness is a feature: it turns silent misaddressing into a loud, debuggable error.

\subsection{attr slots (including boolean attributes and SVG xlink)}

Attribute writes route through \texttt{setAttr(el, name, next)}:

\begin{itemize}
  \item \texttt{null | false} removes the attribute
  \item boolean-ish attributes are present/absent (set to empty string)
  \item SVG \texttt{xlink:*} attributes use \texttt{setAttributeNS}
\end{itemize}

Implementation notes:

\begin{itemize}
  \item SVG detection uses \texttt{namespaceURI} (not \texttt{instanceof SVGElement}) for broader runtime compatibility.
  \item Boolean attribute names are normalized (\texttt{readOnly} maps to \texttt{readonly}).
\end{itemize}

Specs:

\begin{itemize}
  \item \texttt{updates attribute slots by path}
  \item \texttt{treats boolean attributes as present/absent (attr slots)}
  \item \texttt{sets xlink:* attributes with the xlink namespace on SVG elements}
\end{itemize}

\paragraph{DOM ops summary}

\begin{itemize}
  \item \texttt{null | false} \textrightarrow \texttt{removeAttribute} (or \texttt{removeAttributeNS} for SVG xlink)
  \item boolean-ish \textrightarrow \texttt{setAttribute(name, '')} (presence indicates truth)
  \item other values \textrightarrow \texttt{setAttribute(name, String(value))}
\end{itemize}

\subsection{prop slots (input correctness)}

Property slots route through \texttt{setProp(el, name, next)}.

Special-case correctness for inputs:

\begin{itemize}
  \item \texttt{value}: written as a string property, nullish clears to \texttt{""}
  \item \texttt{checked}: written as boolean
\end{itemize}

This matches standard controlled-input behavior: writing to properties updates the live control state, while attributes can drift.

Specs:

\begin{itemize}
  \item \texttt{updates property slots by path}
  \item \texttt{updates input value/checked via property semantics for prop slots}
  \item input control suite in \texttt{block-input-controls.test.ts}
\end{itemize}

\paragraph{Why prop slots exist at all}

Because for form controls the live state is on properties.
Setting an attribute might update the HTML, but not the user-visible control state in the same way.
The tests in \texttt{block-input-controls.test.ts} are basically saying: ``treat blocks like controlled inputs''.

\subsection{class slots}

Class slots write the \texttt{class} attribute:

\begin{itemize}
  \item \texttt{null | false} removes the attribute
  \item arrays are joined with spaces after filtering falsy values
  \item otherwise coerced to string
\end{itemize}

\subsection{style slots}

Style slots accept:

\begin{itemize}
  \item \texttt{null | false}: remove whole style attribute
  \item string: set \texttt{style} attribute directly
  \item object: per-key \texttt{style.setProperty} / \texttt{removeProperty}
\end{itemize}

This small matrix is validated by \texttt{block-slot-matrix.test.ts}.

\subsection{event slots: bind once, swap handler}

Event slots are designed to avoid listener leaks and avoid churn:

\begin{itemize}
  \item the first time a slot is set, the runtime binds a stable listener function
  \item the listener reads \texttt{rec.current} to find the latest handler
  \item updates replace \texttt{rec.current} without re-binding
  \item destroying the block removes the listener
\end{itemize}

Specs:

\begin{itemize}
  \item \texttt{supports event slots: binds once and updates handler without leaking listeners}
  \item \texttt{destroy removes event listeners (no further handler calls)}
  \item \texttt{skips redundant event listener churn when handler identity is unchanged}
\end{itemize}

\subsubsection{Matching Relax VDOM semantics: hostComponent binding}

\texttt{mountBlock(..., \{ hostComponent \})} optionally calls handlers with \texttt{this === hostComponent}, matching Chapter 7’s event binding behavior in the VDOM runtime.

\section{Lifecycle: destroy vs dispose, and idempotency}

\texttt{mountBlock} is not reactive by itself, so \texttt{dispose()} is just an alias for \texttt{destroy()}.

\texttt{destroy()} is idempotent:

\begin{itemize}
  \item it guards with a \texttt{destroyed} flag
  \item it destroys composed children
  \item it removes event listeners
  \item it removes the block root from the DOM
\end{itemize}

After \texttt{destroy()}, \texttt{update()} becomes a no-op (guarded by the same \texttt{destroyed} flag). That prevents stale scheduled work from mutating detached DOM.

\subsection{Why the destroyed guard matters in a scheduled world}

Once you introduce lanes + budgets, "later" becomes a first-class concept.
So the runtime must assume:

\begin{itemize}
  \item a compiled block can be disposed while a flush task is still pending
  \item events may still fire on detached nodes in test environments
\end{itemize}

The destroyed guard plus listener removal turns those situations into safe no-ops.
The disposal tests exist to make this behavior non-negotiable.

Specs:

\begin{itemize}
  \item \texttt{destroy is idempotent even with event slots and children}
  \item \texttt{lifecycle: destroy() is idempotent and prevents further updates}
\end{itemize}

\section{mountCompiledBlock(): wiring signals to slot writes}

\subsection{The contract the compiler relies on}

The compiler's job is to turn template expressions into a list of \texttt{CompiledSlot} entries:

\begin{itemize}
  \item \texttt{key} must match a slot key in \texttt{def.slots}
  \item \texttt{read()} must be a pure read of the current values (it may read signals)
\end{itemize}

The runtime's job is to evaluate these reads inside effects, coalesce them, and commit them through \texttt{block.update()}.

\begin{center}
\begin{tikzpicture}[
  node distance=9mm,
  box/.style={draw, rounded corners, align=left, inner sep=5pt},
  arrow/.style={-Latex, thick}
]

\node[box, right=18mm of read] (pend) {store in \texttt{pending[key]}\\set \texttt{pending=true}};
\node[box, right=18mm of pend] (sched) {schedule once\\lane scheduler};
\node[box, right=18mm of sched] (flush) {flushPending:\\\texttt{batch(() => update(pending))}};

\draw[arrow] (read) -- (pend);
\draw[arrow] (pend) -- (sched);
\draw[arrow] (sched) -- (flush);
\end{tikzpicture}
\end{center}

\texttt{mountCompiledBlock} is the bridge between the reactive graph and the block DOM writer.

Input: \texttt{CompiledSlot[]} where each entry is:

\begin{verbatim}
{ key: string, read: () => unknown }
\end{verbatim}

\textbf{Important:} \texttt{read()} can access signals; that’s how dependency tracking happens.

Mechanism:

\begin{itemize}
  \item compute initial values by running each \texttt{read()}
  \item mount a non-reactive block with those initial values
  \item create one \texttt{createEffect()} per compiled slot
  \item coalesce multiple invalidations into a single scheduled \texttt{flushPending}
  \item execute \texttt{block.update(values)} inside \texttt{batch()} to keep effects consistent
\end{itemize}

The coalescing logic (\texttt{pending} + \texttt{scheduled} flag) is the performance heart: many effects can invalidate in one turn, but only one scheduled task drains \texttt{pending} and commits to the DOM.

Scheduling goes through the deterministic lane scheduler from Chapter 13:

\begin{quote}
	texttt{scheduler.schedule(lane, flushPending)}
\end{quote}

Spec: \texttt{changing a signal updates only the intended DOM node (mountCompiledBlock)}.

\subsection{Why coalescing matters (and how it stays deterministic)}

Without coalescing, a burst of signal writes could schedule and run many DOM commits.
With coalescing:

\begin{itemize}
  \item each effect recomputes its slot value and writes it into a \texttt{pending\{\}} object
  \item a single scheduled task flushes \texttt{pending} to the DOM
\end{itemize}

This mirrors the scheduler philosophy from Chapter 13:
run tasks in a deterministic order, but avoid doing redundant work.

In code form, the shape looks like:

\begin{lstlisting}[language=JavaScript]
// Conceptual shape (names simplified)
const pending = {}
let scheduled = false

function flushPending() {
  scheduled = false
  batch(() => block.update(pending))
  // reset pending
}

createEffect(() => {
  pending[key] = read()
  if (!scheduled) {
    scheduled = true
    scheduler.schedule(lane, flushPending)
  }
})
\end{lstlisting}

The actual implementation is in \texttt{runtime/block.ts} and uses the same primitives you've already learned: \texttt{createEffect}, \texttt{batch}, and the deterministic scheduler lanes.

\subsection{Timeline: what happens when two signals change in the same tick}

The spec \texttt{changing a signal updates only the intended DOM node (mountCompiledBlock)} is tiny, but it implies a timing model.
Here's the practical timeline:

\begin{enumerate}
  \item Two signals are written (often in the same event handler).
  \item Their dependent slot effects re-run and compute new values.
  \item Each effect writes its value into \texttt{pending[key]}.
  \item The first effect that notices \texttt{scheduled === false} schedules \texttt{flushPending} exactly once.
  \item The scheduler later runs \texttt{flushPending} (based on lane + flush strategy).
  \item \texttt{flushPending} calls \texttt{block.update(pending)} inside \texttt{batch()}.
\end{enumerate}

This is the block-runtime equivalent of a "commit phase": compute reactively, then commit a coalesced set of DOM writes.

\subsection{Annotated walkthrough: \texttt{mountCompiledBlock()} line-by-line}

This section is a literate read of the real implementation in \texttt{runtime/block.ts}. The goal is to understand the exact control flow that turns \emph{many reactive invalidations} into \emph{one scheduled DOM commit}.

\begin{lstlisting}[language=JavaScript, caption={The full \texttt{mountCompiledBlock()} implementation (from \texttt{runtime/block.ts})}]
export function mountCompiledBlock(def, host, slots, options = {}) {
  const lane = options.lane ?? 'default'
  const scheduler = options.scheduler ?? createScheduler()

  // initial values from slots
  const initialValues = Object.create(null)
  for (const s of slots) initialValues[s.key] = s.read()

  const block = mountBlock(def, host, initialValues)

  // Coalesce multiple slot invalidations into a single scheduled update.
  let scheduled = false
  const pending = Object.create(null)

  function flushPending() {
    scheduled = false
    const values = Object.assign(Object.create(null), pending)
    for (const k of Object.keys(pending)) delete pending[k]
    batch(() => block.update(values))
  }

  const effects = slots.map((s) =>
    createEffect(() => {
      const next = s.read()
      pending[s.key] = next
      if (scheduled) return
      scheduled = true
      scheduler.schedule(lane, flushPending)
    })
  )

  let disposed = false

  return Object.assign(block, {
    dispose() {
      if (disposed) return
      disposed = true
      for (const e of effects) e.dispose()
      block.destroy()
    },
  })
}
\end{lstlisting}

Now let's go line-by-line and name the invariants.

\paragraph{\texttt{lane} + \texttt{scheduler}: where the commit runs}
\begin{itemize}
  \item \texttt{const lane = options.lane ?? 'default'}: the compiler (or framework integration) chooses the lane that this block uses for DOM commits.
  \item \texttt{const scheduler = options.scheduler ?? createScheduler()}: by default each mounted compiled block gets a scheduler instance; passing one in allows sharing a scheduler across many blocks so their commits interleave deterministically (Chapter~13).
\end{itemize}

\paragraph{Initial read: mount with a consistent snapshot}
\begin{itemize}
  \item \texttt{for (const s of slots) initialValues[s.key] = s.read()}: we read every compiled slot once up front.
  \item This matters because \texttt{mountBlock(...)} immediately calls \texttt{block.update(initialValues)} internally, so the DOM is created already reflecting the current reactive state.
  \item The read happens \emph{before} we install effects, so we don't accidentally schedule any work while mounting.
\end{itemize}

\paragraph{The coalescing state: \texttt{pending} + \texttt{scheduled}}
\begin{itemize}
  \item \texttt{pending}: a plain dictionary \texttt{\{ [slotKey]: nextValue \}} of the latest computed values.
  \item \texttt{scheduled}: a one-bit latch that says whether a flush task has already been enqueued.
\end{itemize}

This pair implements a standard UI-runtime trick: \emph{many producers, single consumer}. Every slot effect can write into \texttt{pending}, but only the first invalidation schedules a consumer run.

\paragraph{\texttt{flushPending()}: the commit phase}
\begin{itemize}
  \item \texttt{scheduled = false}: we open the gate so that new invalidations can schedule the next flush.
  \item \texttt{const values = Object.assign(Object.create(null), pending)}: we take a snapshot of the bag.
  \item \texttt{for (const k of Object.keys(pending)) delete pending[k]}: clear \texttt{pending} in-place.
  \item \texttt{batch(() => block.update(values))}: commit the DOM writes, but keep the signals runtime in a batched mode so downstream effects don't thrash while we patch nodes.
\end{itemize}

Two subtle details here are worth calling out:
\begin{itemize}
  \item \textbf{Snapshot then clear:} if a slot changes \emph{while} we're in \texttt{block.update}, it writes into the cleared \texttt{pending} bag and can schedule a \emph{new} flush (because \texttt{scheduled} was reset first).
  \item \textbf{The update payload is sparse:} \texttt{values} only contains keys that changed since the last flush, so \texttt{block.update} iterates fewer entries.
\end{itemize}

\paragraph{Per-slot effects: producers that only schedule once}
\begin{itemize}
  \item \texttt{createEffect(() => \{ const next = s.read(); ... \})}: each compiled slot becomes an effect.
  \item The effect writes \texttt{pending[s.key] = next} on every re-run.
  \item \texttt{if (scheduled) return}: if a flush is already enqueued, we stop here.
  \item Otherwise we set \texttt{scheduled = true} and enqueue \texttt{flushPending} into the scheduler lane.
\end{itemize}

This is the key guarantee: \emph{no matter how many times effects re-run before the scheduler flushes, we only enqueue one task}. The tasks queue stays small, and the DOM work happens at a predictable point.

\paragraph{Disposal: stop reactive producers, then remove DOM}
\begin{itemize}
  \item \texttt{let disposed = false} + \texttt{if (disposed) return}: disposal is idempotent (safe to call twice).
  \item \texttt{for (const e of effects) e.dispose()}: the critical step. If we destroyed DOM first but kept effects alive, a later signal write would try to update detached (or missing) nodes.
  \item \texttt{block.destroy()}: removes DOM and event listeners (from \texttt{mountBlock}), and also destroys any nested blocks registered via composition helpers.
\end{itemize}

For additional background on why frameworks separate ``compute'' from ``commit'' phases (even when the implementation is small like this), see React's render/commit model \href{https://react.dev/learn/render-and-commit}{https://react.dev/learn/render-and-commit}.

\subsection{Lane choice is part of the compiled contract}

  	\texttt{mountCompiledBlock(..., options)} accepts \texttt{lane} and/or a custom \texttt{scheduler}.
That lets compiled code decide whether DOM writes are:

\begin{itemize}
  \item \textbf{input-latency sensitive} (use \texttt{input} or \texttt{sync}), or
  \item \textbf{deferable} (use \texttt{transition} or \texttt{idle}).
\end{itemize}

So blocks aren't just a faster patching mechanism; they're also a practical location to attach scheduling intent.

That test also checks identity stability:

\begin{itemize}
  \item the static DOM elements are not recreated
  \item text node identity remains stable; only \texttt{nodeValue} changes
\end{itemize}

Spec: \texttt{mounting does not recreate static DOM during updates}.

\texttt{mountCompiledBlock} returns a \texttt{dispose()} method that:

\begin{itemize}

  \item destroys the block DOM
  \item is idempotent
\end{itemize}

\section{Composition: nested blocks and fallback regions}

Blocks are stable and fast, but real UIs still need dynamic structure.
\texttt{runtime/block.ts} provides two small composition helpers:

\begin{itemize}
  \item \texttt{mountNestedBlock(parent, slotKey, def)}: mount a child block into an Element slot
  \item \texttt{mountNestedFallback(parent, slotKey, mount)}: mount a structural fallback region into an Element slot
\end{itemize}

Both helpers:

\begin{itemize}
  \item clear the host element’s existing children
  \item mount into it
  \item register a child destructor so destroying the parent destroys the child
\end{itemize}

Specs in \texttt{runtime/\_\_tests\_\_/block.test.ts}:

\begin{itemize}
  \item \texttt{block composition: can mount a nested block ... and destroys it with parent}
  \item \texttt{block composition: can mount a nested fallback region ... and disposes it with parent}
\end{itemize}

This is HRBR's interop story:

\begin{quote}
Compiled stable subtrees use blocks; dynamic subtrees route to fallback reconciliation.
\end{quote}

Implementation detail: composition is tracked with a small internal child registry.
	texttt{mountBlock} exposes an internal \texttt{\_\_hrbrTrackChild} function, and the composition helpers use it to register child destroyers so teardown order is correct.

\section{Takeaways}

\begin{itemize}
  \item Blocks are "template + slots": a stable DOM instance plus precise write locations.
  \item Slot patching is specialized per kind, with correctness for input controls and SVG namespaces.
  \item Updates skip redundant writes via \texttt{Object.is}.
  \item Event slots bind once and swap handler references; destroy removes listeners.
  \item \texttt{mountCompiledBlock} wires signals to slots and coalesces updates into scheduled flushes.
  \item Composition helpers let blocks nest blocks and fallback regions safely.
\end{itemize}
